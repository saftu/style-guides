SAFTU Style Guide


Table of Contents



    Introduction

    This style guide is meant to be used as the defacto coding standard for all future software development at SAFTU. As of right now (2017) we are migrating our complete codebase to React and React Native. This mainly means our codebase will now purely be in Javascript(and of course HTML + CSS but let's not go into the HTML as a programming language dilemma).


    Source File

    File Name

    File names must all be lowercase. Naming should always be descriptive and words must be separated using underscores (_).

    Strict Mode

    Always use the 'use strict'; directive. The purpose of "use strict" is to indicate that the code should be executed in "strict mode". Strict mode changes previously accepted "bad syntax" into real errors. A comprehensive explanation of the directive can be found here: https://www.w3schools.com/js/js_strict.asp

    Declarations

    Variable Declaration

    • Avoid global variables
    • Always declare local variables (Enforced by the use strict directive).
    • Initialize variables
    • Never declare Number, String, or Boolean Objects. Treat them as primitives.
    • Use const and let. Use const for all of your references, if you must reassign references, use let instead of var.

    Function Declaration

    • Functions should be arranged in alphabetical order
    • Every function must contain header comments following the specified standard.


    Comments

    • Use /** ... */ for multi-line comments.

    • Use // for single line comments. Place single line comments on a newline above the subject of the comment. Put an empty line before the comment unless it's on the first line of a block. Start all comments with a blank space.

    • Use // FIXME: and // TODO: to annotate source files.
    Function Comments

    Special "documentation tags" should be used to give more information. For example, if the function is a constructor, you can indicate this by adding a tag. Parameter and return tags should be used.

    
/**
 * Represents a book.
 * @constructor
 * @param {string} title - The title of the book.
 * @param {string} author - The author of the book.
 */
function Book(title, author) {
}

/**
 * Returns the sum of a and b
 * @param {Number} a
 * @param {Number} b
 * @returns {Number}
 */
function Sum(a, b) {
    return a + b;
}

    Formatting

    Indentation

    Use soft tabs (space character) set to 2 spaces

    
// Good
function blue () {
  return true;
}

// Bad
function blue () {
    return true;
}

    Braces

    Braces are used for all control structures. Braces are required for all control structures (i.e. if, else, for, do, while, as well as any others), even if the body contains only a single statement. The first statement of a non-empty block must begin on its own line.

    
// Bad
if (someVeryLongCondition())
  doSomething();

for (let i = 0; i < length; i++) bar(foo[i]);


// Good
if (someVeryLongCondition()) {
  doSomething();
}

for (let i = 0; i < length; i++) {
  bar(foo[i]);
}

    Spaces Around Operators

    Always put spaces around operators ( = + - * / ), and after commas:

    
var x = y + z;
var values = ["Volvo", "Saab", "Fiat"];

    Line Length

    For readability, avoid lines longer than 80 characters. If a JavaScript statement does not fit on one line, the best place to break it, is after a comma.

    
// Good
this.foo =
    foo(
        firstArg,
        1 + someLongFunctionName());

// Bad
this.foo = foo(firstArg, 1 +
    someLongFunctionName());

    Naming

    Function naming

    Avoid single letter names and be descriptive with your naming. Use PascalCase when naming functions


    
// Bad
function q() {
  // ...
}

function c() {}

// Good
function Query() {
  // ...
}

function ThisIsMyFunction() {}

    Variable and Object naming

    Avoid single letter names and be descriptive with your naming. Use camelCase when naming objects, instances and parameters.

    
// Bad
var OBJEcttsssss = {};

// Good
var thisIsMyObject = {};

    Constant naming

    Constant names use CONSTANT_CASE all uppercase letters, with words separated by underscores.

    
// Constants
const NUMBER = 5;
/** @const */ exports.NAMES = ImmutableList.of('Ed', 'Ann');
/** @enum */ exports.SomeEnum = { ENUM_CONSTANT: 'value' };

// Not constants
let letVariable = 'non-const';
class MyClass { constructor() { /** @const */ this.nonStatic = 'non-static'; } };
/** @type {string} */ MyClass.staticButMutable = 'not @const, can be reassigned';
const /** Set */ mutableCollection = new Set();
const /** ImmutableSet */ mutableElements = ImmutableSet.of(mutable);
const Foo = goog.require('my.Foo');  // mirrors imported name
const logger = log.getLogger('loggers.are.not.immutable');

    Objects

    Use the literal syntax for object creation

    
// Bad
const item = new Object();

// Good
const item = {};

    ESLint

    All of the rules in this document follow standard ESLint style rules. If you're using Visual Studio Code you can install the plugin here(It basically tells you when you mess up):

    Visual Studio Code ESLint

    In order for the plug in to work with our exact ruleset(Extending Airbnb's) your .eslintrc.json must look like this:


    
module.exports = {
  "extends": "airbnb",
  "rules": {
      "react/jsx-filename-extension": ["error", { "extensions": [".js", ".jsx"] }],
      "strict": 0
  },
  "plugins": [
      "react",
      "jsx-a11y",
      "import"
  ]
};
